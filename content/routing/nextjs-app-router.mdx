## Concept explanation

Next.js App Router uses the file system as the router. Every folder inside `app/` represents a URL segment. Special files like `page.tsx`, `layout.tsx`, `loading.tsx`, and `error.tsx` control what renders at each route. This replaces the old `pages/` directory from Next.js 12 and below.

```
app/
в”њв”Ђв”Ђ page.tsx              в†’ /
в”њв”Ђв”Ђ about/
в”‚   в””в”Ђв”Ђ page.tsx          в†’ /about
в”њв”Ђв”Ђ blog/
в”‚   в”њв”Ђв”Ђ page.tsx          в†’ /blog
в”‚   в””в”Ђв”Ђ [slug]/
в”‚       в””в”Ђв”Ђ page.tsx      в†’ /blog/anything
в””в”Ђв”Ђ dashboard/
    в”њв”Ђв”Ђ layout.tsx        в†’ wraps all dashboard pages
    в”њв”Ђв”Ђ page.tsx          в†’ /dashboard
    в””в”Ђв”Ђ settings/
        в””в”Ђв”Ђ page.tsx      в†’ /dashboard/settings
```

## Why it's used

- No manual route configuration вЂ” folder structure is the router
- Nested layouts eliminate repeated code across related pages
- Server Components by default вЂ” pages fetch data without client JS
- Built-in loading, error, and not-found states per route segment

## Real-world use cases

- Marketing site with nested pages (blog, docs, about)
- SaaS dashboard with shared sidebar layout
- E-commerce with product pages (`/products/[id]`)
- Documentation site with category/slug structure
- Multi-tenant app with dynamic subpaths

### Dynamic route вЂ” single param

```tsx
// app/blog/[slug]/page.tsx
type Props = {
  params: Promise<{ slug: string }>;
};

// Generate static pages at build time
export async function generateStaticParams() {
  const posts = await getPosts();
  return posts.map((post) => ({ slug: post.slug }));
}

export async function generateMetadata({ params }: Props) {
  const { slug } = await params;
  const post = await getPostBySlug(slug);
  return { title: post.title, description: post.excerpt };
}

export default async function BlogPostPage({ params }: Props) {
  const { slug } = await params;
  const post = await getPostBySlug(slug);

  if (!post) notFound();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </article>
  );
}
```

### Reading params and searchParams

```tsx
// app/products/page.tsx

  searchParams: Promise<{
    category?: string;
    sort?: string;
    page?: string;

    <div>
      <p>Category: {category ?? "all"}</p>
      <p>Sort: {sort ?? "default"}</p>
      {/* products list */}
    </div>
  );
}
```

## Best practices

- Use route groups `(group)` to apply different layouts to pages at the same level
- Always `await params` and `searchParams` вЂ” they are Promises in Next.js 15+
- Use `generateStaticParams` for dynamic routes that can be pre-rendered at build time
- Use `loading.tsx` per route segment for granular loading states
- Use `error.tsx` per route segment so errors don't crash the whole app
- Use `notFound()` from `next/navigation` inside pages вЂ” never return null for missing data
- Prefer `<Link>` over `router.push()` вЂ” it prefetches on hover and is faster

<Callout type="tip">
  Use route groups to have the home page (`/`) use a different layout than the dashboard (`/dashboard`) without affecting the URLs:

## Interview questions

<InterviewQ question="What is the difference between layout.tsx and template.tsx in Next.js?" difficulty="mid">
  `layout.tsx` persists across route changes вЂ” it mounts once and doesn't re-render when navigating between child routes. This is great for sidebars and headers that should stay mounted. `template.tsx` creates a new instance on every navigation вЂ” it re-mounts and re-renders on each route change. Use `template.tsx` when you need enter/exit animations or need to reset state on every navigation.
</InterviewQ>

<InterviewQ question="What is the difference between [...slug] and [[...slug]]?" difficulty="mid">
  `[...slug]` is a catch-all route вЂ” it matches `/docs/a`, `/docs/a/b` but NOT `/docs` itself (the segment is required). `[[...slug]]` is an optional catch-all вЂ” it matches `/docs` (slug is undefined), `/docs/a`, and `/docs/a/b`. Use optional catch-all when the base route and sub-routes share the same page component.
</InterviewQ>

<InterviewQ question="What is the difference between Server Components and Client Components in Next.js routing?" difficulty="hard">
  By default every component in `app/` is a Server Component вЂ” it runs only on the server, can fetch data directly, and sends zero JS to the browser. Client Components are marked with `"use client"` вЂ” they run in the browser and can use hooks, event handlers, and browser APIs. For routing, `useRouter`, `usePathname`, and `useSearchParams` are client-only hooks. `params` and `searchParams` props work in both server and client components.
</InterviewQ>
