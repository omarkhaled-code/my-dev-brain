## Concept explanation

Zustand is a tiny, fast, and scalable state management library for React. Unlike React Context, which can trigger re-renders across all consumer components when any value changes, Zustand is **subscription-based using selectors**. This means components only re-render when the exact slice of state they are listening to changes.

In Next.js App Router, managing global state requires careful planning because components are Server Components by default. Zustand stores live strictly on the client side. Therefore, we use Zustand to manage highly interactive, ephemeral UI states (like shopping carts, modals, or multi-step wizard forms) while keeping server data fetching decoupled.

<Callout type="note">
  **Next.js vs React difference:** In a plain React SPA, a global singleton
  store (creating a store at the file level) is completely safe. In Next.js,
  because the code executes on the server to handle concurrent user requests, a
  pure global singleton can leak state between different users. For Next.js, we
  must use a custom React Context provider to initialize stores per request.
</Callout>

## Why it's used

- **Zero Boilerplate:** No actions, reducers, or dispatchers like Redux Toolkit; just simple functions to update state.
- **High Performance:** Granular selector system prevents unnecessary component re-renders.
- **Micro Bundle Size:** Weighs around ~1.5KB, ensuring your application bundle stays extremely lightweight.
- **No Provider Inversion:** You can access store actions and state outside of the React render tree if needed.

## Real-world use cases

- **E-commerce Cart:** Updating quantities from product grids and updating the navbar badge instantly.
- **Global UI Framework State:** Managing the open/close state of persistent sidebars, search command palettes (`cmdk`), and dark mode toggles.
- **Canvas or Drag-and-Drop Editors:** Real-time storage of layout coordinates where rapid, high-frequency updates occur.
- **Persistent Multi-step Forms:** Safely caching form drafts locally before submitting them to a Laravel API backend.

## When NOT to Use It

- **Server-Only Data:** If you are just fetching data from a database/API and caching it, use **React Server Components (RSC)** or libraries like **TanStack Query (React Query)** instead of global state.
- **Primitive Component State:** For isolated UI toggles (like `isOpen` for a single dropdown), stick to local `useState`. Do not pollute the global store.

## Code example

### Basic counter store (Client-side only singleton)

```tsx
// store/useCounterStore.ts
import { create } from "zustand";

type CounterState = {
  count: number;
  increment: () => void;
  decrement: () => void;
  reset: () => void;
};

export const useCounterStore = create<CounterState>((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  reset: () => set({ count: 0 }),
}));

### SSR-safe store injection with React Context (Production approach)

// store/todo-store.ts
import { createStore } from "zustand";

export type Todo = { id: string; text: string; completed: boolean };

export type TodoProps = { todos: Todo[] };

export type TodoState = TodoProps & {
  addTodo: (text: string) => void;
  toggleTodo: (id: string) => void;
};

export const createTodoStore = (initProps?: Partial<TodoProps>) => {
  return createStore<TodoState>((set) => ({
    todos: initProps?.todos ?? [],
    addTodo: (text) =>
      set((state) => ({
        todos: [...state.todos, { id: crypto.randomUUID(), text, completed: false }],
      })),
    toggleTodo: (id) =>
      set((state) => ({
        todos: state.todos.map((t) => (t.id === id ? { ...t, completed: !t.completed } : t)),
      })),
  }));
};

// store/todo-provider.tsx
"use client";

import { createContext, useContext, useRef, type ReactNode } from "react";
import { useStore } from "zustand";
import { createTodoStore, type TodoState } from "./todo-store";

export type TodoStoreApi = ReturnType<typeof createTodoStore>;
export const TodoStoreContext = createContext<TodoStoreApi | undefined>(undefined);

export function TodoStoreProvider({ children, initialTodos = [] }: { children: ReactNode; initialTodos?: any[] }) {
  const storeRef = useRef<TodoStoreApi>(null);
  if (!storeRef.current) {
    storeRef.current = createTodoStore({ todos: initialTodos });
  }

  return (
    <TodoStoreContext.Provider value={storeRef.current}>
      {children}
    </TodoStoreContext.Provider>
  );
}

export function useTodoStore<T>(selector: (store: TodoState) => T): T {
  const context = useContext(TodoStoreContext);
  if (!context) throw new Error("useTodoStore must be used within a TodoStoreProvider");
  return useStore(context, selector);
}
```

```tsx
// app/todos/page.tsx — Server Component
import { TodoStoreProvider } from "@/store/todo-provider";
import { TodoList } from "@/components/TodoList";

async function fetchTodosFromDB() {
  // Simulating a database fetch (e.g., SQLite/MySQL via Laravel or Direct)
  return [
    { id: "1", text: "Learn Next.js 16 Architecture", completed: false },
    { id: "2", text: "Configure Zustand Global States", completed: true },
  ];

export default async function TodosPage() {
  const initialTodos = await fetchTodosFromDB();

  return (
    <div className="p-8 max-w-xl mx-auto space-y-6">

      </TodoStoreProvider>
```

## Common mistakes

<WrongRight>
<Wrong>
```tsx
// This causes the component to re-render when ANY state in the store changes!
const { count, text } = useCounterStore()
```
</Wrong>
<Right>
```tsx
// Highly optimized: component only re-renders if 'count' changes.
const count = useCounterStore((state) => state.count)
const text = useCounterStore((state) => state.text)
```
</Right>
</WrongRight>

<WrongRight>
<Wrong>
```tsx
const user = useUserStore((state) => state.user)
const updateName = () => {
  user.name = "Alex" // Direct mutation won't trigger re-render!
}

````
</Wrong>
<Right>
```tsx
// Inside your store creation logic:
updateName: (newName) => set((state) => ({
  user: { ...state.user, name: newName }
}))
````

</Right>
</WrongRight>

Gotchas from production
When using standard Zustand stores with Next.js, you might encounter Hydration Errors if the client-side state dynamically defaults to values from localStorage while the server builds the page with default values. Always handle hydration with a simple useEffect check or a separate mounted state.

## Best practices

**Community conventions**

- One Store Per Feature: Don't put everything into one monolithic store. Separate stores into useAuthStore, useCartStore, etc.
- Always use selectors: Always grab specific values using the analytical selector pattern useStore(state => state.value).

**Performance tips**

- Use shallow middleware if you are returning an object from a selector to prevent unnecessary re-renders when keys don't change.

**Security considerations**

- Never store sensitive user tokens, passwords, or personal data inside an unencrypted client-side store, especially if utilizing persistent middleware like localStorage.

<Callout type="tip">
  Wrap `PaginationURL` in a `Suspense` boundary when using `useSearchParams` — Next.js requires it to avoid blocking the page render:
```tsx
  import { Suspense } from "react"

  <Suspense fallback={null}>
    <PaginationURL ... />
  </Suspense>
```
</Callout>

## Interview questions

<InterviewQ
  question="Why does Zustand not require a Context Provider wrapper around the app tree like Redux or React Context?"
  difficulty="mid"
>
  Zustand stores are external vanilla JavaScript structures. The store holds
  state references outside the React Virtual DOM entirely. When you invoke the
  custom hook, Zustand internally registers the host component using a native
  subscription listener mechanism, updating the component selectively via
  localized state changes only when requested keys shift value.
</InterviewQ>

<InterviewQ
  question="How does Zustand avoid unnecessary component re-renders that frequently bottleneck standard React Context integrations?"
  difficulty="mid">
  Unlike React Context which forces an update pass across all rendering trees descending beneath a modified Provider reference, Zustand depends on structural selectors. Components strictly request localized keys (state => state.target). Zustand checks differences using strict object equality (===) and targets triggers to that matching viewport exclusively.
</InterviewQ>
<InterviewQ
  question="What structural problem can happen if you declare a dynamic global store singleton on Next.js Server Components, and how do you fix it?"
  difficulty="hard">
  Global singletons on pure server files preserve static memory spaces across different concurrent network calls. This means unrelated terminal clients could accidentally access or update cross-user dynamic state values. You fix it by isolating client stores using the "use client" flag at the top of the file, or using factories inside custom React Context definitions if scoped server context mapping is required.
</InterviewQ>

## My notes

<MyNotes>
  Be extremely careful with Zustand middlewares like `persist`. If I update the
  store's TypeScript type definitions or schema structure in production, old
  users will have broken states saved in their local storage. Always bump up the
  `version` configuration inside the persist options block to clear outdated
  client-side storage states cleanly!
</MyNotes>

## Reusable Code Snippets

```tsx
import { create } from "zustand";
import { persist } from "zustand/middleware";

interface AuthState {
  token: string | null;
  user: { id: string; email: string; name: string } | null;
  setAuth: (token: string, user: AuthState["user"]) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      token: null,
      user: null,
      setAuth: (token, user) => set({ token, user }),
      logout: () => set({ token: null, user: null }),
    }),
    {
      name: "auth-session",
    },
  ),
);
```
